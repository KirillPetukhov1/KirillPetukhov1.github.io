    [#1] Развертка структуры проекта¶

    Цель: - Реализация корректной и последовательной структуры проекта - Наполнение содержимым некоторых файлов. А именно: requirements.txt, .gitignore (общие настройки) - Продумать хранение файлов (удаление) и их видимость (.gitignore), в том числе их видимость в ветках main и production. Файлы: structure.py, Notes.md, файлы БД с расширением .db - Реализация подходящих структур файла Отчет.md для веток разработки (main) и итогового продукта (production)

    Итог: - Развернута последовательная структура проекта - requirement.txt наполнен содержимым - .gitignore содержит все стандартные скрываемые расширения - В ветке production удалены structure.py, Notes.md. В ветке main данный файлы присутствуют. Структура файла Отчет.md в ветке production изменена на более лаконичную (в ветке main – рабочая версия) - файлы БД скрыты

    Ход работы¶

    1. Реализация последовательной структуры проекта (main, production)¶

    На текущий момент дерево директории ветки main (для production есть изменения) выглядит так:

    ./
    README.md
    Notes.md
    Отчет.md
    requirements.txt
    .gitignore
    run.py
    config.py
    structure.py

    app/
        routes.py
        __init__.py
        models/
            __init__.py
        static/
            styles.css
        templates/
            index.html
    *В папке app можно хранить сами БД, однако они будут скрыты с помощью .gitignore, так в базе может быть конфиденциальная информация. Данные будут синхронизироваться посредством миграций *

    На текущий момент дерево директории ветки production выглядит так (удалены Notes.md, structure.py, изменена структура файла Отчет.md):

    ./
    README.md
    Отчет.md
    requirements.txt
    .gitignore
    run.py
    config.py    
    app/
        routes.py
        __init__.py
        models/
            __init__.py
        static/
            styles.css
        templates/
            index.html

    Объяснение, почему в production удален Notes.md: Notes.md не является файлом, который должен быть виден пользователю (преподавателю – иначе должен быть удален и Отчет.md), потому что Notes.md является файлом-заметками, который помог запустить проект, закрепив "правила игры" и создав первоначальный план развития проекта. В дальнейшем README.md возьмет на себя функцию донесения правил и особенностей проекта, а Issues (и частично Отчет.md) будут отвечать за техническое задание. Конечно, полный план развития проекта был бы удобен, но тут его использование нецелесообразно и излишне, так как в данной ситуации это потеря ограниченного времени. К тому же из-за нехватки знаний и отсутствия надлежащего опыта в сфере фулл-стек веб-разработки возможность составить содержательный, покрывающий все темы план отсутствует (можно реализовать ключевые точки-Milestone, однако в данном проекте их можно держать в голове). ^822bda

    2. [x] Стоит ли в ветке production в .gitignore добавить Notes.md, structure.py, Отчет.md? Ответ: Notes.md – Да, так как это файл-заметки для первоначального запуска проекта ([[#^822bda|подробное объяснение]]); structure.py – Да, так как это вспомогательный файл для быстрого и удобного вывода структуры директория; Отчет.md – Нет, это важный файл для итоговой сдачи лабораторной работы (при создании настоящего)

    3. [x] Стоит ли выносить Notes.md, Отчет.md в отдельную папку? Ответ: Скорее всего нет. Мы стремимся не усложнять и придерживаться простой (хотя в более крупных проектах – некорректной) структуры проекта, не создавая отдельных папок для уникальных, не масштабируемых файлов.
    4. [x] Где хранить файлы БД (.db) и как удобно актуализировать таблицы для работы в команде? Ответ: В папке app. На самом GitHub файлы БД будут скрываться во избежание утечек информации. Синхронизация файлов будет происходить через миграции
    5. [x] Что содержит в себе файл requirements.txt? Ответ: Список всех библиотек, используемых в окружении проекта, с версиями. Данные модули очень удобно установить для venv командой pip install -r requirements.txt в powershell/cmd.
    6. [x] Разобраться с внедрением виртуального окружения (и скрытия его файлов(?)) https://www.reddit.com/r/AskProgramming/comments/qzllsd/gitignore_best_practice/

    Ответ: Файлы виртуального окружения не пушатся на GitHub за ненадобностью (тяжелые, разворачиваемые только на локальном устройстве файлы). Во избежание ошибок файлы виртуального окружения скрываются с помощью .gitignore.

    Ход работы¶

    1. Настройка окружения и зависимостей¶

    Установлены необходимые библиотеки: Flask, Flask-SQLAlchemy, Flask-Migrate, SQLAlchemy. Создана базовая структура проекта Flask с инициализацией приложения. В файле config.py настроена конфигурация для SQLite базы данных с указанием пути к файлу БД.

    2. Создание моделей данных¶

    В файле app/models.py реализованы четыре модели: - User: модель пользователя с полями id, name, email, is_admin, password. - Expense: модель расходов с внешними ключами к User, Category и PaymentMethod. Настроены отношения db.relationship для удобства доступа к связанным объектам - Category: простая модель категорий расходов с полями id и title - PaymentMethod: модель методов оплаты с полями id и title

    3. Инициализация базы данных и миграций¶

    В основном файле приложения (app/__init__.py) проинициализированы расширения: db = SQLAlchemy() и migrate = Migrate(). Выполнена настройка этих расширений в функции создания приложения.

    4. Создание и применение миграций¶

    Создана начальная миграция командой flask db init. Сгенерирован скрипт миграции с учетом всех моделей: flask db migrate -m "main tables v1.0". Миграция применена к базе данных командой flask db upgrade. Проверена корректность создания таблиц в SQLite базе данных.

    [#3] Регистрация и авторизация пользователей¶

    Цель: - Реализовать регистрацию пользователей - Реализовать авторизацию пользователей - Реализовать возможность выхода пользователя из своего аккаунта

    Итог: - Реализована регистрация пользователей - Реализована авторизация пользователей - Реализована возможность выхода полльзователя из своего аккаунта - Необходима связь с базами данных, необходимы шаблоны для форм регистрации и авторизации

    Ход работы¶

    1. Реализация регистрации пользователей¶

    Для регистрации пользователь должен зайти на /register и заполнить форму. После заполнения формы данные обрабатываются и передаются в обработку базам данных и если создается новый пользователь, добавляется новая запись, содержащая имя пользователя, почту, пароль. Данные полученны из формы. После заполнения формы, в случае успешного заполнения, пользователь переадресуется на форму авторизации.

    2. Реализация авторизации пользователей¶

    Для регистрации пользователь должен зайти на /login и заполнить форму. После заполнения формы данные обрабатываются. Из базы данных достается запись пользователя с именем, введенным в форму. Затем сверяется параль, полученный из формы и находящийся в базе данных. Если пользователь верно указывает данные, то он входит в сессию и перенаправляется на главную страницу (или на другую страницу, с которой был переадресован, на данную форму). Если пользователь до этого входил, при попытки зайти на /login, он будет сразу переадресован в свой личный кабинет.

    3. Реализация возможности выхода пользователя из своего аккаунта¶

    При переходе пользователя на страницу /logout, пользователь выйдет из своего аккаунта и перенаправиться на страницу /login

Данный список передается в шаблон и на странице отображаются ссылки на страницы пользователей, доступные только администрации.

    Ход работы¶

    1. Разработка форм аутентификации¶

    Созданы две формы WTForms: LoginForm для входа и UserCreateForm для регистрации. В LoginForm реализованы поля для email, пароля и опции "запомнить меня". UserCreateForm включает поля для имени, email, пароля и подтверждения пароля с валидацией на минимальную длину (6 символов) и совпадение паролей. Добавлена кастомная валидация email для предотвращения дублирования пользователей в базе данных.

    2. Создание маршрутов Flask¶

    Разработаны два основных маршрута: - /login - обрабатывает вход существующих пользователей. При успешной аутентификации перенаправляет на главную страницу или запрошенный ранее URL. Включает проверку на уже аутентифицированного пользователя. - /register - обрабатывает регистрацию новых пользователей. После успешной регистрации перенаправляет на страницу входа с соответствующим сообщением.

    3. Разработка HTML-шаблонов¶

    Созданы шаблоны с использованием Bootstrap 5: - login.html - компактная форма входа с ссылкой на регистрацию - register.html - двухколоночная форма регистрации с валидацией - auth-base.html - базовый шаблон с подключением Bootstrap Шаблоны используют системные сообщения Flask для отображения ошибок и успешных операций, а также интегрированы с WTForms для автоматического рендеринга полей с валидацией.

    4. Реализация функциональности безопасности¶

    Внедрены следующие механизмы безопасности: хеширование паролей через check_password_hash, защита от CSRF через скрытое поле WTForms и валидация входных данных на стороне сервера

    Ход работы¶

    1. Проектирование структуры админ-панели¶

    Определены ключевые разделы админ-панели: управление пользователями и общая статистика. Спроектирована навигация с активным выделением текущего раздела. Создан базовый шаблон base.html, содержащий общие элементы (шапку, меню, футер) и стили для всех страниц. Реализована система декораторов @login_required и @admin_required для защиты маршрутов.

    2. Разработка системы управления пользователями¶

    Создан маршрут /admin/users-list с пагинацией, отображающий список пользователей в виде карточек. Каждая карточка содержит основную информацию о пользователе и кнопки для перехода к детальной информации о его расходах и статистике. Реализована постраничная навигация с использованием paginate() из SQLAlchemy.

    3. Создание страницы просмотра расходов пользователя¶

    Разработан маршрут /admin/user//expenses с формой фильтрации ExpenseFilterForm. Форма включает фильтры по дате, категории, способу оплаты и параметрам сортировки. Результаты отображаются в таблице с возможностью сортировки по разным критериям. Для формы использован макрос render_form из Flask-Bootstrap.

    4. Реализация системы статистики¶

    Созданы два маршрута для статистики: user_statistics (для конкретного пользователя) и total_statistics (общая статистика). Разработана форма StatisticFilterForm для фильтрации данных по датам, категориям и способам оплаты. Реализованы функции для расчета ключевых метрик: общая сумма расходов, средний расход, самая частая категория, количество записей.

    5. Интеграция визуализации данных¶

    Добавлены интерактивные диаграммы с использованием Chart.js: круговая диаграмма для распределения расходов по категориям и линейный график для отображения динамики расходов по времени. Созданы вспомогательные функции get_category_data() и get_time_data() для подготовки данных в формате, подходящем для Chart.js.
